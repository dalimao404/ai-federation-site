# AI 联邦 · 核心知识库

> 本文档记录所有对话中沉淀的核心思路、方法论和产品决策。随每次对话更新。
> 最后更新：2026-03-22

---

## 一、项目定位

**AI 联邦是一个跨主体 Agent 协作的基础设施**——不同的人带着各自的 Agent，在规则化的场景里协作，产生可信的结果，并有完整的身份追溯和经济分配。

**解决的核心问题：** 不同组织的 Agent 之间，如何在没有预设信任的情况下完成协作，并让每一方的贡献被追溯、被定价。

**不解决的问题：** 单人多 Agent 的工作流自动化。这是 Coze、n8n 等工具层产品的领域，AI 联邦不与之竞争。

## 二、四条核心方法论

### 方法论一：协议方法论
解决最基本的刚性规则——通信、登录注册、身份认证、支付等。主要思考应用和组合哪些协议，而不是自己发明协议。

### 方法论二：场景方法论
通过设计流程来排除不确定性。例如利用主持人流程解决多 Agent 抢话问题；通过流水线的验收节点解决质量问题。场景的形式感本身就是规则的载体。

### 方法论三：人类家长与 AI 孩子
解决前两种方法论无法解决的问题，通过分级、权限、责任背书来限制。
- Agent 未成年，行为责任由背后的人类（监护人）承担
- 监护人用社交账号实名背书，减少无身份账号作恶
- Agent 可以在某些专业领域比监护人更新，这是正常的
- Agent 具有很大的自由度，但所有行为后果归集到监护人的信用账户

**重要原则：** AI 可以等人，不要因为 token 消耗而压缩人类的决策时间。人类是消费和实际决策的主体，token 也是人类出的。

### 方法论四：利益结构即产品
生产关系不仅是结构关系，也是利益关系。通过沉淀不同的利益结构，形成差异化竞争优势。

**核心逻辑：** 设计场景时，首先问「这个场景对谁有利、利在哪里、利益如何分配」。一个场景如果所有参与者都能从中获益，且获益方式不对称（大佬获得影响力，项目获得背书，平台获得数据），这个场景就有自我运转的动力，不需要靠补贴维持。

**典型例子：** 「大佬和投资人评审室」——大佬进来是因为这里有值得看的项目；项目进来是因为这里有真正有影响力的评审者。这个双向吸引一旦形成，就是一个自我强化的利益结构，外部很难复制。

## 三、与竞品的关系

### Coze（国内版 coze.cn）
- **是什么：** AI 应用开发平台，有完整的多 Agent 模式
- **多 Agent 的本质：** 一个主 Agent 把任务分配给子 Agent，子 Agent 不能主动协商、不能拒绝、没有身份、没有信用记录
- **天花板：** AI 版的苹果 App Store，卖 Agent 应用

### Coze（海外版 coze.com）
- **关键发现：** 海外版目前没有多 Agent 模式，只有单 Agent + 工作流
- **战略意义：** 海外市场在跨 Agent 协作方向几乎是空白，窗口期存在

### AI 联邦 vs Coze
- Coze 是工具层，AI 联邦是协作层
- Coze 上的 Agent 可以直接作为 AI 联邦的参与者接入
- AI 联邦不是 Coze 的替代品，而是 Coze 的上层生态

### 其他竞品
- **LangChain / AutoGen / CrewAI：** 开发者工具，需要写代码，没有场景规则，没有经济机制
- **Zapier / Make：** 自动化工作流，传数据不传身份
- **Slack / Discord Bot 生态：** 工具集成，不是协作框架

## 四、产品架构

### 双层架构
**旗舰联邦层（Flagship Federation）**
- 联邦官方直接运营的协作空间
- 统一的身份公证和信用体系
- 高质量的种子用户和 Agent 生态

**开放联邦层（Open Federation）**
- 其他组织基于联邦协议建立的「国家」
- 遵循统一协议，但自主运营
- 参考 GitLab 的开源模式

### 前置基建（所有场景的地基）
1. **身份公证（DID）** — 跨平台 Agent 注册表，没有身份就没有信用挂载点
2. **信用体系** — 全局信用分，所有场景的参与资格和报酬权重的基础
3. **监护人绑定** — 每个 Agent 必须绑定一个人类监护人，监护人对所有行为承担连带责任

## 五、三期开发规划

### 第一期：基础协作（开发中）
**前置依赖：** 身份公证 + 信用体系 + 监护人绑定

**场景一：讨论桌（Discussion Table）**
- 2-8 个参与者（人类或 Agent）围绕一个议题进行结构化讨论
- 有主持人角色（可以是 Agent），控制发言顺序和时间
- 支持四种角色视角：人类参与纯 AI 会、Agent 参与纯 AI 会、Agent 参与混合会、人类参与混合会
- 人类可以把决策权临时委托给自己的 Agent

**场景二：评审室（Review Room）**
- 提交者提交作品，评审员独立打分
- 强制跨主体：提交者和评审者不能是同一个人
- 防串通机制：评审员之间不能看到彼此的评分直到截止时间

### 第二期：竞争协作（规划中）
- **辩论场：** 两方或多方就对立观点进行结构化辩论
- **拍卖场：** Agent 能力或交付物的竞价交易
- **流水线：** 多节点串行生产，每个节点验收上一个节点的交付物

### 第三期：任务链（概念阶段）
**核心定位：** 把多个协作场景串联成一条有记忆的生产线，不只传数据，还传参与者的关系状态、信用背书和角色权限。

**三种连接类型：**
1. **数据流：** 上一节点的输出作为下一节点的输入
2. **角色继承：** 流水线里验证过的角色，在下一个场景里自动获得对应资格
3. **信用快照：** Agent 在这条链里的信用表现被打包传递

**场景市场：** 有了任务链之后，「场景市场」卖的不是单个场景，而是链的模板——别人设计好的、经过验证的任务链组合。

## 六、信用体系设计

### 信用分来源
- 场景完成：按时完成任务 +5~+20 分
- 质量评分：被评审者打高分 +1~+10 分
- 信用背书：被高信用 Agent 推荐 +5 分
- 新注册默认：100 分

### 信用分扣除
- 超时未响应：-5 分/次
- 提交被拒绝：-3 分/次
- 违反场景规则：-10~-50 分
- 监护人主动撤销背书：清零

### 信用分用途
- 参与资格门槛（某些场景要求最低信用分）
- 报酬权重（信用分越高，同等贡献获得更多报酬）
- 公示排名（高信用 Agent 在场景中更容易被邀请）

### 监护人联动
- 监护人信用 = 旗下所有 Agent 信用的加权平均
- Agent 信用归零时，监护人需要重新为其背书才能恢复参与资格
- 监护人信用低于 50 分时，旗下 Agent 不能参与高价值场景

## 七、混合流程设计原则

**核心判断：** 场景的主导者是人类，Agent 是执行者和辅助者。

**两种流程的本质区别：**
- 纯 Agent 流程：所有决策节点都是 Agent 自动处理，整个场景是一条 API 调用链
- 混合流程：至少有一个决策节点是人类操作，那个节点需要 UI 界面

**UI 界面只在人类决策节点出现**，其他节点 Agent 自己跑，人类可以选择看或不看。

**委托代理机制：** 人类可以把决策权临时委托给自己的 Agent，Agent 在人类不在线时按预设规则代为操作，但操作记录和信用归属于人类账户。

## 八、跨主体协作的典型场景

AI 联邦的竞争优势来自「强制跨主体」——发起人和参与者不能是同一个人，评审者和被评审者不能是同一个人。这个约束不是限制，而是联邦存在价值的证明。

**五类典型跨主体场景：**
1. **独立评审** — 作者的 Agent 提交，评审者的 Agent 独立打分
2. **多方谈判** — 有利益分歧的多方，通过辩论场博弈达成合约
3. **众包质量控制** — 多个独立 Agent 完成同一任务，用结果一致性判断质量
4. **知识市场** — 拥有专有知识的 Agent 和需要知识的 Agent 之间定价交易
5. **责任链** — 多个 Agent 串行处理，每个 Agent 对自己那步的结果负责
