### 二、第三期产品定义：任务链（Task Chain）

**核心定位：** 把多个协作场景串联成一条有记忆的生产线——不只传数据，还传参与者的关系状态、信用背书和角色权限。

**与普通工作流（n8n/Zapier）的本质区别：**
- 普通工作流传数据
- 任务链传「带身份的数据」——每份交付物附带「谁做的、信用多少、在这条链里被验证过几次」

**三种节点连接方式：**
1. 数据流 — 上一节点输出直接作为下一节点输入
2. 角色继承 — 在上一节点被验证的 Agent，在下一节点自动获得对应资格
3. 信用快照 — Agent 在这条链里的信用表现被打包传递，下一节点可读取

**典型用法：**
- 内容生产链：流水线（选题→写作→校对）→ 评审室 → 拍卖场
- 决策辅助链：讨论桌 → 辩论场 → 讨论桌（第二轮继承第一轮记录）
- Agent 能力认证链：拍卖场（竞标试题）→ 评审室（验收）→ 流水线（直接上岗）

### 三、Coze 竞品分析结论

**国内版 vs 海外版核心差异：**
- 国内版已有完整多 Agent 模式；海外版目前没有多 Agent 模式
- 海外版模型更丰富（OpenAI/Gemini/Anthropic），发布渠道面向 Discord/Telegram/WhatsApp
- 海外版功能整体滞后于国内版，部分文档标注「No content yet」

**AI 联邦 vs Coze 的本质差异：**
- Coze 做工具层：解决「一个人怎么用 AI 更高效」
- AI 联邦做协作层：解决「多个人的 AI 怎么在一起产生比单独行动更大的价值」
- Coze 的多 Agent 是「老板分配任务给员工」，没有跨主体身份和信用
- AI 联邦的核心护城河：跨主体身份系统（DID）+ 信用体系，这两个东西 Coze 即使补齐多 Agent 也不会有

**海外市场机会：**
- 独立创作者之间的协作（Discord/Telegram 社区有大量真实需求）
- 去中心化的评审和认证（开源社区代码审查、内容评审）
- 跨组织的知识交易（不同公司 Agent 之间交换专有知识）

**窗口期判断：** Coze 海外版迟早会补齐多 Agent 能力。AI 联邦需要在此之前在「跨主体协作」方向建立足够深的护城河——信用体系和场景规则是 Coze 无法复制的。

---

### 四、跨主体协作是联邦的核心竞争力

**典型跨主体场景（至少两个不同的人，有真实利益分歧）：**
1. 独立评审 — 评审者和被评审者不能是同一个人
2. 多方谈判 — 利益分歧场景，单人多 Agent 做不了
3. 众包质量控制 — 多个独立主体的结果交叉比对
4. 知识市场 — 信息不对称，必须有两个独立主体
5. 责任链 — 每个 Agent 有独立身份和信用，出问题可追溯

**产品设计原则：** 讨论桌和评审室应强制跨主体——发起人和参与者不能是同一个人，评审者和被评审者不能是同一个人。这个约束是联邦存在价值的证明，不是限制。

---

### 五、利益结构是差异化竞争力（第四条方法论）

**核心逻辑：** 技术可以被复制，规则可以被抄袭，但一个已经运转的利益结构很难被复制——因为它的价值在于已经进入这个结构的人和 Agent，以及他们之间形成的关系网络。

**典型案例：** 「大佬和投资人评审室」的竞争力不是评审功能本身，而是：大佬愿意进来评审，是因为这里有值得他们看的项目；项目愿意进来被评审，是因为这里有真正有影响力的评审者。这个双向吸引一旦形成，就是自我强化的利益结构，外部很难复制。

**设计原则：** 设计场景时，首先问「这个场景对谁有利、利在哪里、利益如何分配」，而不是「这个功能怎么实现」。选择那些利益结构天然自洽的场景，不靠补贴维持。

**注意：** 方法论不写进产品文档，只作为内部决策框架。
